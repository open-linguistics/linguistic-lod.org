Dear all,
I have struggled for quite some time with trying to understand how the LLOD
cloud is supposed to be used. So I guess my proposal would be to select
data sets for inclusion based on their relevance for the relevant use
cases. E.g. if the LLOD cloud is supposed to be a curated, aggregated data
set like bio2rdf, used by linguists to do research, only data sets which
can be automatically integrated into it would qualify (and AFAIKT this
would exclude quite a few data sets) - and inclusion could be handled on a
case by case basis upon request. If on the other hand the LLOD cloud is a
collection of metadata mainly used to generate the LLOD cloud image, other
selection criteria may be more appropriate.
best
robert

>
> As discussed today in the telco, we plan to come to terms with this
> definition as soon as possible, so, please have a look on the proposed
> definitions for "linguistically relevant" datasets, add your own or express
> your opinion about an existing one before *the next telco* (Feb 18th, 12:15
> CET). At this telco, we will discuss a decision procedure to implemented
> thereafter.
>
> As a reminder, the goal is to come up with an *operational* definition
> that provides criteria to justify the inclusion of non-linguistic, but
> important data sets such as DBpedia or Yago into the LLOD cloud, but to
> eliminate clearly irrelevant resources, like, say, biomedical data bases.
